Behavioral Patterns

Chain of Responsibility (CoR)

Description

The CoR pattern allows for more than one object in a hierarchy of objects the chance to receive and handle a message send.  The objects are arranged like a “chain” and if an object can’t handle a message send, the message is passed along the “chain” until it can be handled.  In a lot of cases, the chain can be terminated using a combination of the Singleton and Null Object patterns.

Forces

The CoR pattern should be used in a hierarchy of objects when the sender needs an answer, regardless of who it comes from.  For example, a sender may ask object A for its description, and if A can’t handle the request, it is sent up the “chain of command” to A’s successor.  This process continues until the message can be handled.

Solution

The top level object in a “chain of command” can handle the message request, but children of the top level object may or may not be able to handle the same message.

In the following example, getDescription() isn’t understood by the Child class, so it is passed up the “chain of command” to the Parent class.

public class Parent {

    public String description;

   

    public Parent() {

        this.description = "Default description.";

    }

   

    public String getDescription() {

        return this.description;

    }

}

public class Child extends Parent {

    public Child() {

        super();

    }

}

public class ChainOfResponsibility {

    public static void main(String[] args) {

        Child c = new Child();

        System.out.println(c.getDescription());

    }

}

Running the ChainOfResponsibility program will output the default description even though the message is not understood by the Child class.

Output:

Default description.

Null Object

Description

The Null Object pattern is an object that replaces NULL, nil, or null.  Rather than an invalid object of type A equalling null (and resulting in program crashing) it equals the null object.  The null object can be a singleton and is an instance of A.  The Null Object represents the absence of an object.

Forces

The Null Object pattern can be used in any scenario where an object may not be instantiated.  For example, an array of type A objects may not allow two instances of A with the same name.  If instance anA has a the same name of an existing object is the array, anA would normally be set to null.  In this case, any messages send to anA would crash the program.  Instead, if anA is set to the null object, all messages will be understood and will be handled appropriately.

Solution

In the following example objects of type Apple are added to an ArrayList.  If an apple of the specified kind already exists in the ArrayList, it isn’t added and a NullApple object is returned.

public class NullApple extends Apple {

    public NullApple(String kind) {

        super(kind);

    }

   

    public String getKind() {

        System.out.println("Accessing instance of NullApple.");

        return "NULL";

    }

}

public class Apple {

    private String kind;

   

    public Apple(String k) {

        this.kind = k;

    }

   

    public String getKind() {

        return this.kind;

    }

}

import java.util.*;

public class NullObject {

    private static ArrayList<Apple> apples = new ArrayList();

   

    public static void main(String[] args) {

        System.out.println(addApple("Gala").getKind());

        System.out.println(addApple("Granny Smith").getKind());

        System.out.println(addApple("Gala").getKind());

    }

   

    // returns the newly added apple

    // return the null object if the apple wasn't added

    private static Apple addApple(String kind) {

        Apple result = null;

        if (!appleExists(kind)) {

            result = new Apple(kind);

            apples.add(new Apple(kind));

            return result;

        } else

            result = new NullApple(kind);

        return result;

    }

   

    // check to see if an apple of "kind" already exists in the list

    private static boolean appleExists(String kind) {

        for (int i = 0; i < apples.size(); i++)

            if (apples.get(i).getKind().equals(kind))

                return true;

        return false;

    }

}

Running the NullObject program above will add successfully add Gala and Granny Smith apples to the list and fail when trying to add the second Gala apple.

Output:

Galla

Granny Smith

Accessing instance of NullApple.

Command

Description

The Command pattern is used to encapsulate a request in an object using an abstract Command class.  This allows the request to passed around and stored as if it were an object.  This Command class includes an interface for executing requests.

Forces

The Command pattern is the object-oriented replacement for callbacks.  It should be used when requests are to be executed at different times.  For example, a program implementing the undo or redo functions would store a command that performs the requested operation when the user presses the undo or redo button.

Solution

The solution to the Command pattern is to create an abstract Command class that implements an interface for request execution.  This interface specifies an Execute method.  Subclasses of the Command class invoke this Execute method in combination with an instance variable for the receiver.

Once this command object is created it can be passed around and invoked at any time.

Memento

Description

The Memento pattern is most commonly used for undo and redo operations, but it also exists in system restore and backup.  The Memento pattern encapsulates versions of an object so they can restored to a previous version some time in the future.

Forces

Depending on what needs to be done, sometimes it is necessary to store the current state of an object.  For example, for an undo command, previous operations need to be remembered so when the undo is performed, the object can easily be reset back to a previous state.

Similarly, older version of a word document are automatically stored in case of a system crash.  In this case, the backed up document will be okay if the current one suddenly became corrupt.

Solution

An object that adopts the Memento pattern implements a private instance variable that stores its current state.  It also implements the ability to create new mementos and set the current state.  This allows the object to be easily restored to any of its mementos with minimal effort.

Double Dispatch

Description

The Double Dispatch pattern is used to narrow a problem down to a simpler form by passing the receiver as an argument.  It is a relay message from the sender to the receiver and back to the receiver.

Forces

The Double Dispatch pattern is extremely useful when some command needs to be executed depending on the receiver’s type, which isn’t known.  Double Dispatch is used in the Visitor pattern when traversing through a list of objects of known types.

Solution

See the Visitor Pattern.

Visitor

Description

The Visitor pattern allows different operations to be performed on an object without changing the internal structure of that object.  For example, a set of data may need to be exported in different file formats - HTML, JSON, or raw text.  In this case, a new visitor can be implemented for each file format without having to change the internal structure of the data model.

Forces

The Visitor pattern should be used for traversing over an object structure without knowing the type of each object.  For example, an array contains several different fruits, but it doesn’t know the kind of each fruit.  The Visitor pattern is used to answer that question.  It allows the receiver to visit each object and perform a different operation for each type.

Solution

The key method to the Visitor pattern is the accept method.  The accept method is implemented in each object that is to be traversed.  This method then relays a message back to the Visitor object saying, “Hey! Here I am and this is my type.” This allows the Visitor to perform different operations for each type.

In the following example, Apple and Orange implement the Visitable interface which allows a Visitor object to visit them.  With the Visitor pattern, an array can easily be traversed, executing different code depending on the type of the object in the array.

public interface Visitable {

    public void accept(Visitor aVisitor);

}

public class Apple implements Visitable {

    private String color;

   

    public Apple() {

        color = "Green";

    }

   

    public String getColor() {

        return this.color;

    }

   

    public void accept(Visitor aVisitor) {

        aVisitor.visitApple(this);

    }

}

public class Orange implements Visitable {

    private String name;

   

    public Orange() {

        name = "Orange";

    }

   

    public String getName() {

        return this.name;

    }

   

    public void accept(Visitor aVisitor) {

        aVisitor.visitOrange(this);

    }

}

public class Visitor {

    public Visitor() {

       

    }

   

    public static void visitApple(Apple a) {

        System.out.println("Apple color: " + a.getColor());

    }

   

    public static void visitOrange(Orange o) {

        System.out.println("Orange name: " + o.getName());

    }

}

import java.util.*;

public class VisitorExample {

    public static void main(String[] args) {

        Visitor v = new Visitor();

        ArrayList<Visitable> fruit = new ArrayList<Visitable>();

        fruit.add(new Orange());

        fruit.add(new Apple());

       

        for (Visitable f: fruit)

            f.accept(v);

    }

}

Running the VisitorExample program will traverse through the list of fruit and execute a different Visitor method depending on the type of fruit.

Output:

Orange name: Orange

Apple color: Green

Intention Revealing Message

Description

The Intention Revealing Message pattern is one of the simplest patterns that should be adopted by all programmers.  It involves giving your functions, methods, variables, etc. names that have meaning in the context they are used.

A name should be clear and understandable to anyone who might read the code later.

Forces

In the production world there are many people who work on the same project and contribute to the same code, not only in present time, but in the future as well.  Names that have meaning are much more likely to be properly interpreted in the future by new developers.

Intention revealing names and messages make a project easier to understand, read, update, and maintain.

Solution

In the following example it is clear which is the better choice: addIntegers has an intention revealing name whereas a could mean anything.

public class IntentionRevealingExample {

    public static void main(String[] args) {

        System.out.println(addIntegers(5, 3));

    }

   

    // good

    public static int addIntegers(int a, int b) {

        return a + b;

    }

   

    // bad

    public static int a(int a, int b) {

        return a + b;

    }

}

Output:

8

Template Method

Description

The Template Method pattern is used to describe a general purpose method or class that can be used in more than one context.  This method or class is dependent on factors that differ between contexts.

Forces

The Template Method pattern should be used when code is repeated more than once.  In this case, the similarities in code is used to create a new method or class.  The differences in code are implemented as instance variables or method parameters, and affect how the method or class works.

Solution

It is extremely important to keep code updated, and template methods or classes make that process much easier.  Refactoring of code should be done periodically to strengthen its readability and maintainability.

In the following example common code is refactored into a method that can be called from anywhere with different arguments.

import java.util.*;

public class TemplateMethod {

    public static ArrayList<Integer> ints = new ArrayList<Integer>();

   

    public static void main(String[] args) {

        addInteger(5);

        addInteger(10);

        System.out.println(ints);

    }

   

    public static void addInteger(int intToAdd) {

        for (Integer i: ints)

            if (i == intToAdd)

                return;

        ints.add(intToAdd);

    }

}

Instead of writing a separate loop every time an integer is added, it is written once and placed in a method that can be called multiple times.  This makes the code both more readable and more maintainable.

Output:

[5, 10]

Observer

Description

The Observer pattern allows a set of observers to be notified when an object changes.  This is especially useful in user interfaces when several components need to be refreshed if, for example, the top level data object is updated.

Forces

The Observer pattern can be used in the following situations:

  1. When a change to an object requires changing multiple other objects.
  2. When an object needs to notify other objects without knowing how those objects work.

The important part of the Observer pattern is that the object sending notifications (i.e. the object being observed) doesn’t know anything about its observers, yet it can communicate to them when it is changed or updated.

Solution

An object, A, stores a collection of dependencies.  Each dependency is an observer and is automatically notified when the A updates.  Each observer implements a method that is executed when A updates.

This enables a dynamic layered architecture: a model that can be transferred between operating systems.

Coordinator

Description

The Coordinator pattern is used to mediate interactions between objects.  It is most commonly used in a layered architecture such as the following:

User Interface

Coordinator

Application Model

In this case, the Coordinator manages communication between the UI and Application Model.  The UI and Application Model cannot communicate directly.

Forces

The Coordinator pattern is great for applications that aim to support multiple platforms.  Since the UI won’t have direct access to the application model, the same model can be used across platforms.  This makes the backend of the application much easier to manage and maintain without having to worry about how it may affect the UI.

Solution

The Coordinator object knows how to communicate with both the User Interface and the Application Model.  When it receives a message for either layer it knows how to relay that message to the correct receiver.

Iterator

Description

The Iterator pattern provides a way to access sequential objects without exposing its internal structure.

Forces

The Iterator pattern can be used when more than one traversal method is required.  It is useful when accessing an object’s contents because it doesn’t expose its internal structure.  It also provides an interface for traversing different kinds of structures.

Solution

The key element in the Iterator pattern is having an Iterator class that is able to handle all aspects of a traversal.  It stores the current item, the previous item, the next item, and it can determine whether or not the traversal is done.  

The Iterator class is a template class that can be utilized in many different data structures.

Collecting Parameter

Description

The Collecting Parameter is used in most cases to collect operational data from many different sources.  These sources can be anything from objects to methods.  A pointer to an object is passed to a method and the method records its actions in the passed object.

Forces

At times, it is necessary to accumulate the results of the actions of many different objects or methods.

Solution

In the following example, an object is passed as a parameter to a method that records the result of that method.  In this case, true is added if total is greater than 100, false if it is not.

import java.util.*;

public class CollectingParameter {

    private static Integer total = new Integer(1);

   

    public static void main(String[] args) {

        ArrayList<Boolean> results = new ArrayList<>();

       

        int c = 0;

        while (c < 8) {

            doubleTotal(results);

            c++;

        }

       

        for (Boolean b: results)

            System.out.print(b + " ");

    }

   

    public static void doubleTotal(ArrayList<Boolean> results) {

        total += total;

        if (total > 100)

            results.add(true);

        else

            results.add(false);

    }

}

Output:

false false false false false false true true

Structural Patterns

Composite

Description

The Composite pattern allows a hierarchy of objects to be treated as if there were a single instance of an object.  

Forces

The Composite pattern is used in a hierarchical tree structure of objects, and each subtree needs to be treated as it’s own object.  In this situation, any node of the tree can be thought of as the root node of it’s leaves, treating it as it’s own object.

Solution

The solution of the Composite pattern is to implement a common protocol among tree nodes.  This way, internal nodes can delegate any requests to other parts of the tree if needed.

In the following example, root is of type Node and it’s children are of type Leaf, but because both Leaf and Node implement the CompositeInterface they are able to understand the same messages.  This allows each element in the tree to be treated as its own object.

public interface CompositeInterface {

    public int getValue();

    public CompositeInterface getRight();

    public CompositeInterface getLeft();

}

public class Leaf implements CompositeInterface {

    private int value;

    private CompositeInterface right;

    private CompositeInterface left;

   

    public Leaf(int val, CompositeInterface leftNode, CompositeInterface rightNode) {

        value = val;

        left = leftNode;

        right = rightNode;

    }

   

    public int getValue() {

        return this.value;

    }

   

    public CompositeInterface getRight() {

        return this.right;

    }

   

    public CompositeInterface getLeft() {

        return this.left;

    }

}

public class Node implements CompositeInterface {

    private int value;

    private CompositeInterface right;

    private CompositeInterface left;

   

    public Node(int val, CompositeInterface leftNode, CompositeInterface rightNode) {

        value = val;

        left = leftNode;

        right = rightNode;

    }

   

    public int getValue() {

        return this.value;

    }

   

    public CompositeInterface getRight() {

        return this.right;

    }

   

    public CompositeInterface getLeft() {

        return this.left;

    }

}

public class Composite {

    public static void main(String[] args) {

        Leaf left = new Leaf(5, null, null);

        Leaf right = new Leaf(15, null, null);

        Node root = new Node(10, left, right);

       

        System.out.println(root.getLeft().getValue());

        System.out.println(root.getLeft().getValue());

        System.out.println(root.getValue());

    }

}

Running the Composite program will create a tree structure of three nodes and print the value at each node.

Output:

5

5

10

Facade

Description

The Facade pattern uses a top level interface that is applied to smaller, more specific interfaces in a structure of subsystems.  For example, a library’s data store model may have a top level interface for a general piece of work, and then subinterfaces for types of work such as video, journal, novel, etc.

In the diagram above, each “block” is it’s own interface that implements the parent interface.  Using the Facade pattern greatly reduces complexity in a large system.

Forces

The Facade pattern should be used to make complex systems simpler or when several dependencies that share common attributes are present in a system.  The Facade pattern is also a great way to implement communication between the different levels of a system.

Solution

See the Composite pattern solution.

Creational Patterns

Singleton

Description

The Singleton pattern is used when one instance and only one instance of a class is needed.  For example, a singleton object may be used in combination with the Null Object pattern and the Chain of Responsibility pattern to terminate the end of a linked list.

Forces

The Singleton pattern should be used when exactly one instance of a class is needed, and when this instance needs to be globally accessed.

Solution

The solution to the Singleton pattern is to treat the class like any other.  The difference is the class’s constructor methods return the unique instance of the class rather than creating a new one.  The very first time a constructor is called, a unique instance of the class is created.  This is the instance that is accessed later.

public class StorageManager {

    private static StorageManager sharedManager = null;

    public Integer numberOfFiles = new Integer(0);

   

    protected StorageManager() {

       

    }

   

    // only create a new instance if it hasn't already been initialized

    public static StorageManager getSharedManager() {

        if (sharedManager == null)

            sharedManager = new StorageManager();

        return sharedManager;

    }

}

public class Singleton {

    public static void main(String[] args) {

        StorageManager m = StorageManager.getSharedManager();

        m.numberOfFiles += 5;

       

        StorageManager m2 = StorageManager.getSharedManager();

       

        // even though they are different variables, they are pointed to the

        // same instance of StorageManager

        System.out.println(m.numberOfFiles);

        System.out.println(m2.numberOfFiles);

    }

}

Running the Singleton program above will modify the global instance of StorageManager and print the number of files from two difference variables that are pointed to the same instance of StorageManager.

Output:

5

5
